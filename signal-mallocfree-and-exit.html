signal, malloc/free and exit

Date Sat 03 January 2015 Tags signal / deadlock / exit

最近处理了一个bug，系统在特定情况下处理信号出现了死锁。 在这篇文章里，我会描述一下遇到的问题，解决方案和由此产生的一些思考

背景

这是一个多进程的系统，每个进程都只有一个线程工作。有一个主进程负责处理用户连接，创建子进程并将工作交给子进程处理。 进程之间通过共享内存共享一些数据和状态。 共享内存由锁保护形成临界区。 任何进程在临界区里发生错误，主进程都会向所有子进程发送SIGQUIT，导致所有子进程进入信号处理函数并最终调用exit()退出。 在所有子进程退出后，主进程会重新恢复共享内存并等待新的连接。 问题发生在系统内存不足的情况下。由于大量涌现的并发连接，系统内存被耗尽。 一个子进程在临界区内发生了OOM并抛出了错误。 主进程向所有子进程发出了SIGQUIT但是若干子进程产生了死锁并未退出。 最终导致整个系统无法恢复。

分析问题

我们先来分析一下进程死锁时的状态。

从调用栈我们可以看出，死锁进程在收到信号的时候正在尝试分配内存。 信号处理函数最终调用exit()函数希望退出进程并通知父进程，但是 exit()函数触发了C++静态对象的析构函数。 对象析构时尝试delete/free内部成员。由于malloc/free都不是信号安全的函数，malloc/free内部实现中会去加同一个锁，导致单x线程死锁发生。 问题的根源在于信号处理函数中调用了非信号安全的函数exit()。 关于信号安全的函数可以参考这个链接。 分析清楚问题产生的原因之后，解决问题就变得简单了，使用_exit()代替exit()即可。

exit()如何结束一个进程？

从上面的分析我们不禁会产生好奇，为什么_exit()是信号安全的， 但exit()不是？ 这要从exit()如何结束一个进程说起。在exit()终止一个进程之前，要按顺序完成以下4件事情。

  1. 调用由atexit()/on_exit()注册的回调函数。
  2. 刷新所有打开的输出文件流。
  3. 关闭所有打开的文件描述符。
  4. 删除所有由 tmpfile()创建的临时文件。

由于exit()需要调用由用户注册的回调函数，因此exit()不可能保证信号安全。 而与exit不同，exit()只执行前面列出的第2，3，4项，不调用用户注册的回调函数，因此能保证信号安全。 _exit()与C++ 如果没有使用atexit()注册任何回调函数，exit()能否保证信号安全呢？ 答案是否定的。在我们之前遇到的问题中，我们实际上并没有注册任何的回调函数，但是死锁还是发生了。 细心的人会发现这与C++对象的析构有关系。让我们先做个实验。 

class Test {
public:
    Test() {
        cout << "ctor of Test" << endl;
    }

    ~Test() {
        cout << "dtor of Test" << endl;
    }
};

以上是一段用来测试的代码，Test类在构造函数和析构函数中分别打印出不同的字符串。 我们首先测试一下exit()函数。 

static Test test;

int main() {
    cout << "!!!Hello World!!!" << endl;
    exit(2);
}


ctor of Test
!!!Hello World!!!
dtor of Test

我们再来测试一下_exit()函数。 

static Test test;

int main() {
    cout << "!!!Hello World!!!" << endl;
    _exit(2);
}


ctor of Test
!!!Hello World!!!

通过这个实验，我们能够发现exit()与_exit()的另外一个区别是是否调用C++静态对象的析构函数，这个链接有详细的说明。 exit()与_exit()的一个显著的区别在于前者是一个C库函数定义在头文件<stdlib.h>/<cstdlib>中，后一个是POSIX标准系统调用，定义在<unistd.h>中。 因此_exit()并不关心语言的具体细节，因此也不会去析构任何静态对象了。
